# STEER-001 Product Steering (Purpose / Non-goals / Success Metrics)

## LLM_BRIEF

- kind: steering（scope: spec_system）
- iori-spec の**プロダクト指針（目的・非目的・成功指標）**を定義する。
- 本仕様は、要件（REQ）や境界（IF/DATA/TEST）を設計する際の意思決定の“上位基準”を提供する（詳細は各 REQ に委譲）。
- 技術方針（設計原則・互換性方針）は STEER-002 を正とし、本仕様は **「何のために作るか／何を作らないか／どう成功を測るか」** に集中する。

## このドキュメントの役割

### 何を決めるか（本仕様の SSOT）

本仕様（STEER-001）は、iori-spec のプロダクトとしての方向性を一貫させるために、以下を SSOT として定義する。

- プロジェクトの狙い（Purpose）／対象ユーザー／提供価値（Value Proposition）
- 非目的（Non-goals）とスコープ境界（何を「しない」か、何を「別レイヤーに委譲」するか）
- 成功指標（定量KPI＋定性）と、優先順位付けの原則
- 主要ユースケースのサマリ（詳細な要件・境界定義は REQ/IF/DATA/TEST に委譲）

### 何を決めないか（委譲先）

- 具体的な機能要件・受入条件（Acceptance Criteria）：REQ-* が SSOT
- 境界仕様（API/CLI/イベント/ジョブ等）：IF-* が SSOT
- データ契約（スキーマ・互換性の詳細）：DATA-* が SSOT
- テスト観点・合否基準：TEST-* が SSOT
- 仕様書システムの運用詳細：SPEC-SYS-001 が SSOT
- セクション定義とスキーマ：SPEC-SYS-002 が SSOT
- トレーサビリティ規約：SPEC-SYS-003 が SSOT
- 成果物契約（index/pack/lint_report の shape・互換性・決定性）：SPEC-SYS-004 が SSOT
- ツール実行仕様・CIゲート：SPEC-SYS-005 が SSOT

### 本仕様が答える主要な問い（チェックリスト）

- iori-spec は「誰の、どんな困りごと」を解決するのか（対象ユーザーと課題）
- “成功”を何で測るのか（定量・定性の指標と測定方法）
- 作らないもの／後回しにするものは何か（非目的と境界）
- 仕様体系（REQ/IF/DATA/TEST）を増やすとき、何を優先するか（優先順位の原則）
- 主要ユースケースは何で、どの REQ 群に分解されるべきか（導線）

## 範囲（Scope）と前提（Assumptions）

### 対象（In scope）

- iori-spec の“プロダクトとしての指針”（意思決定の基準）
- 仕様書群（Markdown）とツール（index/lint/pack/trace/CI gate）を一体として運用する前提の、ユーザー価値・非目的・成功指標
- 小〜中規模開発（個人〜少人数チーム）における SDD（Specification-Driven Development）＋LLM 協働を主対象とする

### 非対象（Out of scope）

- 個別機能の仕様本文（REQ/IF/DATA/TEST の詳細）
- 具体的なUIデザイン、ブランド、販売戦略、価格などのビジネス詳細（必要なら別途 steering に切り出す）
- 特定の実装技術スタックの固定（技術方針は STEER-002 側で管理）

### 前提（Assumptions）

- 仕様は「人間にもLLMにも読める」ことが重要であり、索引化・局所コンテキスト化（pack）・トレーサビリティ（trace）を用いて更新容易性を担保する。
- “Stable Core” を中心に、互換性・決定性・SSOT を守りながら拡張していく（詳細は STEER-002 / SPEC-SYS-004）。
- LLM は「全体を常時記憶する存在」ではないため、局所推論導線（LLM_BRIEF / READ_NEXT / pack）を設計の前提とする。

### 用語・定義（Definitions）

- **SDD**：仕様（Spec）を SSOT として設計・実装・検証を進める開発運用。
- **SSOT**：Single Source of Truth。正とする定義の所在。
- **pack**：LLM に渡す局所コンテキストのパッケージ（抽出範囲は `include_in: pack` に従う）。
- **trace**：REQ/IF/DATA/TEST 等の論理的つながり（意味論は SPEC-SYS-003 を正とする）。

## プロダクトの狙い（Purpose）と提供価値

- 仕様書群が増えても「迷わない・壊れない」状態を維持し、LLM を活用した設計・実装・レビューを安定運用できるようにする。
- 仕様の **検索性（index）／整合性（lint/trace-lint）／局所推論可能性（pack）** を、ツールと規約で担保する。
- 人間の意思決定（設計の論点）と、LLM の作業（生成・整合確認）を分離し、変更に強い開発ループを作る。

## 対象ユーザー

- 仕様駆動で開発したい個人開発者・小規模チーム
- LLM と協働しつつ、仕様の整合性・トレーサビリティを崩さずに進めたい開発者
- 仕様が肥大化して「どれが正か」「どこを更新すべきか」が分からなくなる問題に直面しているチーム

## 非目的（Non-goals）とスコープ境界

- 仕様を書けば自動で“正しい実装が完成する”ことを保証しない（実装・テストは依然として責務分割が必要）。
- 仕様管理のためのSaaSやホスティング、権限管理・ワークフローエンジン等の提供は当面の対象外。
- 仕様の内容（プロダクト要件）そのものの妥当性を自動で証明することは目標にしない（本ツールは整合性・形式・導線を主に担保する）。
- すべての設計論点を 1 ファイルに集約しない（原子化・トレーサビリティ・局所コンテキスト化を優先）。

## 成功指標（Success Metrics）と優先順位の原則

### 成功指標（候補）

定量（例）：

- 仕様の更新に伴う **lint / trace-lint の重大違反（ERROR）** が、PR時点で検知されリリース前に残らないこと。
- “局所コンテキスト（pack）”のみで、指定タスク（要件整理/IF更新/テスト生成等）を遂行できる割合が継続的に改善すること。
- 新規参画者が「主要導線（LLM_BRIEF/READ_NEXT）」だけで、目的の仕様に到達できるまでの時間が短縮されること。

定性（例）：

- 「どれが正か」「どこを直すべきか」の迷いが減り、仕様更新の心理的コストが下がること。
- LLM に渡す情報が過不足なくなり、生成結果の手戻りが減ること。

※指標の確定（数値目標・計測手段）は、運用成熟に応じて別途 steering/metrics として切り出してよい。

### 優先順位の原則

- **正（SSOT）の明確さ**と**変更容易性**を最優先する（“どこが正か分からない”状態を許容しない）。
- 仕様の **決定性（determinism）** と **互換性（compatibility）** を損なう変更は慎重に扱う（詳細は STEER-002 / SPEC-SYS-004）。
- 仕様は「読む人/LLMが迷わない導線」を持つべきであり、網羅性よりも **局所推論可能性** を優先する。

## 主要ユースケース（サマリ）

- 新規仕様（REQ/IF/DATA/TEST）の追加：テンプレに従って作成し、trace を結線し、pack で局所推論可能にする。
- 仕様の修正：影響範囲を特定し（trace/依存）、関連 spec を同一PRで更新し、lint/trace-lint でゲートする。
- 参照（設計・実装・レビュー）：LLM_BRIEF→READ_NEXT→pack の順で読み進め、必要最小限の文脈で作業する。

## USAGE

- 想定読者（Who）:
  - 仕様体系の設計者（PdM/Tech Lead/アーキテクト）、仕様を書く開発者、レビュー担当者
- 参照トリガー（When）:
  - 仕様体系の立ち上げ時（最初の REQ 群を作る前）
  - 仕様体系の運用ルールを変更する時（命名・粒度・breaking/deprecationの扱いなど）
  - 仕様群が増えた/迷子が増えた/手戻りが増えたとき（成功指標の再点検）
- 使い方（How）:
  - 新規 REQ を切る前に、本仕様の目的/非目的/成功指標に照らして粒度と優先度を確認する
  - 仕様の運用ルールを検討する際に、何を守るべきか（SSOT/導線/互換性）を上位基準として参照する
  - 指標（lint/trace-lint、pack運用）に問題が出た場合の、方針の見直しに用いる
- セット読み（With）:
  - STEER-002 — 技術指針（Stable Core 境界・互換性方針）
  - SPEC-SYS-001 — 仕様書群の書き方・運用フロー（実務手順）
  - SPEC-SYS-004 — 成果物契約（決定性・互換性の基準）

## 運用上の目安（LLM / SDD 観点）

- 更新トリガー（Trigger → Action）:
  - 対象ユーザーや提供価値が変わった／焦点を変える必要が出た:
    - Action: 本仕様を更新し、影響する STEER-002 および主要 REQ 群（優先順位）を再点検する
  - 仕様運用上の痛み（迷子・更新漏れ・手戻り）が顕在化した:
    - Action: 成功指標（どの指標で検知するか）と優先順位原則を見直し、SPEC-SYS-001/005 のゲート運用も点検する
  - 互換性・決定性に関する扱いを変更する必要が出た:
    - Action: 原則は STEER-002 / SPEC-SYS-004 を更新し、本仕様は“プロダクト指針としての影響”を反映する
- LLM 連携（貼り方）:
  - 最小セット: 本仕様の LLM_BRIEF + 変更したい観点（目的/非目的/指標）+ 期待成果物（更新案/差分）
  - 拡張セット: 関連する STEER-002 と、問題の発生箇所（lint/traceの失敗例、迷子の具体例）を追加で渡す
- ツール運用（Gate）:
  - 方針の変更は、関連 spec（STEER/SPEC-SYS/主要REQ）を同一PRで更新し、lint/trace-lint を必須で通す
